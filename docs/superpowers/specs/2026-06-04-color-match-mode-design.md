# 颜色匹配模式设计

## 目标

为 Bounce Buddy 增加颜色匹配玩法，同时保持现有三轨跳跃控制简单直接。玩家需要观察当前小球颜色，选择有效落点，并利用彩虹落点切换颜色、延续路线。

## 核心规则

- 每局开始时，小球颜色为红色。
- 每个落点都有一种类型：普通颜色落点或彩虹落点。
- 普通颜色落点只有在颜色与当前小球颜色一致时才有效。
- 彩虹落点对任意小球颜色都有效。
- 落到彩虹落点后，小球会变成该彩虹落点指定的新颜色。
- 落错轨道，或落到颜色不匹配的普通落点，都会结束游戏。
- 只有有效落点才加分；有效的彩虹落点也会加分。

## 颜色

普通玩法颜色使用三种：

- 红色：小球初始颜色。
- 蓝色。
- 黄色。

彩虹落点使用彩虹或白色高亮的视觉表现，让玩家理解它是“任意颜色可落”的特殊点，而不是第四种普通颜色。每个彩虹落点内部仍会记录一个“下一颜色”。

## 平台生成

保留现有轨道与距离系统：

- 三条横向轨道。
- 每个落点之间保持固定距离。
- 每个落点序号只显示 1-2 个平台，保持画面稀疏和选择压力。
- 平台宽度收窄，让不同轨道的平台在视觉上保持明确分隔。

颜色生成增加一层稳定路线规则：

- 每个落点第一次生成时就固定平台数量、轨道、颜色和类型，后续不再临时改写已经出现的平台。
- 路线按落点序号同时记录颜色和主轨道；主轨道每次最多移动一格，保证一次切轨可达。
- 普通落点组至少包含一个与当前路线颜色匹配的平台。
- 部分落点组会把其中一个平台替换为彩虹落点，作为必须换色点。
- 彩虹落点被踩中后，小球颜色变为该彩虹落点指定的新颜色。
- 彩虹落点的下一颜色使用稳定循环：红色 → 蓝色 → 黄色 → 红色。
- 每局开始时，默认目标轨道应自动指向下一跳中与红球匹配的红色落点，避免开局就是危险选择。
- 生成序列本身应该始终保留至少一个与当前小球颜色匹配的前进选择，且不通过临场变色来兜底。

## 视觉设计

- 小球材质颜色和自发光颜色反映当前小球颜色。
- 普通落点使用自己的玩法颜色作为主体和发光颜色。
- 彩虹落点使用更亮的混合色或彩虹效果，并带更强的边缘光。
- 现有目标提示环继续用于表示当前选择是否有效：
  - 青色表示选中的下一落点在轨道和颜色上都有效。
  - 粉色表示选中的下一落点会因为轨道或颜色不匹配而失败。
- HUD 提示文案应简短说明新规则：`只能落到同色平台；彩虹平台会换色`。

## 失败反馈

失败时，提示文案说明原因：

- 没有落到平台：`没有落到平台`。
- 颜色不匹配：`颜色不匹配`。

游戏结束、最终得分和最佳分数逻辑保持不变。

## 实现备注

- 功能继续集中在 `src/main.js`，不引入新框架或复杂抽象。
- 增加一个小型颜色配置对象，集中维护小球、平台颜色和中文标签。
- 将 `platformLayout` 的值从单个横向坐标扩展为对象，包含轨道位置、平台类型、颜色和可选的下一颜色。
- 保留现有速度选择、键盘切轨、计分和最佳分数存储行为。

## 验证方式

如果 Vite 开发服务已经运行，直接刷新 `http://127.0.0.1:5173/` 并检查：

- 每局开始时小球为红色。
- 小球为红色时，可以有效落到红色平台。
- 落到颜色不匹配的普通平台会结束游戏，并显示颜色失败提示。
- 彩虹平台接受任意小球颜色，并在有效落地后改变小球颜色。
- 分数、最佳分数、重新开始、速度选择和轨道切换仍然正常。

除非用户明确要求，不主动运行 `npm run build`。
